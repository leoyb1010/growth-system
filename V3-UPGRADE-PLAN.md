# 增长业务工作台 · V3 升级方案

> 从"模块化数据管理后台"升级为"面向管理动作的团队业务工作台"

---

## 一、当前系统问题诊断

### 产品维度

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | **导航按"表结构"组织**，不是按"管理动作"组织。用户进来看到的是"核心指标/重点工作/月度工作/季度成果"——这是数据库表名，不是管理者关心的事 | 高 |
| 2 | **无周/日节奏感**。系统完全没有"本周"和"今天"的概念，所有页面默认展示"当前季度全部数据"，管理者3分钟看不出重点 | 高 |
| 3 | **无角色区分**。管理员和部门负责人看到完全一样的首页，信息优先级不匹配 | 中 |
| 4 | **录入和查看分离**。要更新进度必须跳到"数据录入"页或打开编辑弹窗，无法在项目列表内就地更新 | 高 |
| 5 | **缺乏闭环**。项目只记录"本周进展"，不强制要求"下一步动作"；风险项没有"需谁决策/何时收口" | 中 |

### UI/结构维度

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | Dashboard 信息密度低。4张KPI卡+2个图表+2个列表，但管理者最关心的"今天变了什么、本周必须做什么"看不出来 | 高 |
| 2 | 列表只支持按更新时间排序，不支持按风险度/重要度/是否临期排序 | 中 |
| 3 | 卡片信息不完整。项目卡看不到"上次更新时间""距截止还几天""是否有阻塞" | 中 |
| 4 | 周报是纯数据拼装，没有"管理结论"——只告诉你"哪些项目风险了"，不说"建议怎么做" | 中 |

### 逻辑维度

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | 状态不统一。Project用"未启动/进行中/风险/完成"，Performance用"正常/预警/严重"，MonthlyTask又用"未启动/进行中/风险/完成"——语义不同 | 中 |
| 2 | 无"今日更新"概念。所有数据只看季度，没有按天/按周聚合的视图 | 高 |
| 3 | 后端已有 is_risk / severe_warning / is_due_soon 计算，但前端没有充分利用这些信号驱动排序和展示 | 低 |

## 二、升级后的产品定位

**"团队周管理与日更新工作台"**

不是录入+查询+导出系统，而是——

1. **管理者**打开系统，3分钟内知道本周和今天最该关注什么
2. **负责人**能够低成本按天更新，按周复盘
3. 数据、状态、风险、动作、结果形成闭环
4. 周报是管理表达，不是数据拼接

│ └──────┘ └──────┘ └──────┘ └──────┘            │
├─────────────────────────────────────────────────┤
│ 今日变化 + 本周关注（2列）                         │
│ ┌─────────────────┐ ┌─────────────────┐        │
│ │📋 今日变化       │ │⚡ 本周关注       │        │

### 关键新增逻辑

1. **"今日变化"区块**：后端新增 `GET /api/dashboard/today-changes`，返回今天有更新的记录（状态变更、进度更新、新增项），前端展示为时间线
2. **"本周关注"区块**：后端新增 `GET /api/dashboard/week-focus`，返回本周需收口项、偏差较大指标、长期未更新项，规则驱动
3. **"待收口事项数"**：`status !== '完成' && due_date <= 本周末` 的项目计数

### 后端新增接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/dashboard/today-changes` | GET | 今日数据变更列表（读AuditLog） |
| `/api/dashboard/week-focus` | GET | 本周关注点（规则驱动） |
| `/api/dashboard/week-summary` | GET | 本周摘要统计 |

**实现方式**：利用现有 `updated_at` 字段 + AuditLog 表，不做新表。

**现状问题**：KPI 和业绩分两个页面，管理者关心的是"目标达成情况"，不是"KPI表"和"业绩表"

**风险**：低。合并展示逻辑，不删原有页面

## 七、日更新链路设计

### 核心思路
管理者不需要专门去"数据录入"页更新，而是在日常查看中顺手更新。

### 实现方式

**1. 项目卡就地更新**
```
点击进度条 → 弹出 InputNumber → 输入新进度 → 自动保存
点击"本周进展"文字 → 弹出 TextArea → 输入 → 自动保存
```

**2. "本周"页面的日更新流**
```
/week 页面结构：

┌──────────────────────────────────────┐
│ 📅 本周管理  4/21 ~ 4/27             │
├──────────────────────────────────────┤
│ 今日动态（今天有什么变化）             │
│ - XX项目 进度 60%→75%               │
│ - YY项目 状态 进行中→风险            │
├──────────────────────────────────────┤
│ 今日待更新（哪些项需要今天补录）       │
│ - 拓展组·渠道合作项目 本周进展未更新   │
│   [快速更新] → 弹出 TextArea         │
├──────────────────────────────────────┤
│ 今日到期                             │
│ - 3个项目今天到期，2个未完成           │
├──────────────────────────────────────┤
│ 本周重点工作列表（可就地编辑）          │
└──────────────────────────────────────┘
```

**3. 后端支持**
- `GET /api/dashboard/today-changes`：今日变更
- `GET /api/projects/stale`：超过3天未更新的项目（需补充更新）
- `PUT /api/projects/:id/quick-update`：快速更新接口（只更新 weekly_progress / progress_pct / status）

### 风险说明

| 改动 | 风险 | 说明 |
|------|------|------|
| 就地更新交互 | 低 | 新增API，不改原有编辑接口 |
| 今日动态 | 低 | 读取AuditLog，不写新表 |
| 待更新提醒 | 低 | 利用updated_at字段计算 |

## 八、周管理链路设计

### 核心思路
"本周"是一个一等公民概念，不是一个筛选条件。

### 本周页面的完整结构

```
/week

Tab1: 📋 本周概览
  - 本周经营摘要（GMV/利润完成率 + 风险项数 + 待收口数）
  - 本周重点工作列表（默认只看本周有更新的）
  - 风险项目清单
  - 7天内到期清单

Tab2: 📝 今日更新
  - 今日变更流
  - 今日待更新项（一键快速更新）
  - 今日到期提醒

Tab3: ⚡ 下周重点
  - 下周重点工作列表（现有ProjectPage的nextweek Tab逻辑）
```

### 周管理时间计算

```javascript
// 本周 = 本周一00:00 ~ 本周日23:59
const getThisWeekRange = () => {
  const today = moment();
  const dayOfWeek = today.isoWeekday();
  const monday = today.clone().subtract(dayOfWeek - 1, 'days').startOf('day');
  const sunday = monday.clone().add(6, 'days').endOf('day');
  return { start: monday, end: sunday };
};
```

### 风险说明

| 改动 | 风险 | 说明 |
|------|------|------|
| 新增 /week 路由 | 低 | 新页面，不替换旧页面 |
| 后端时间范围查询 | 低 | 在现有查询条件加 updated_at 范围过滤 |

## 九、周报重构方案

### 现状问题
周报只是5段数据表格拼接，没有管理结论。

### 改法

**1. 周报结构升级**

```
增长组业务周报 | 4/21~4/27

【本周结论】（新增）
  整体完成率78%，低于时间进度（83%），需加速。
  拓展组GMV偏差最大，建议XX重点关注。

【关键变化】（新增）
  · 运营组利润完成率从65%提升到72%
  · 渠道合作项目状态变更为风险

一、本周数据摘要（保留）
二、重点工作进展（保留，增加"下周动作"列）
三、风险与预警（保留，增加"建议处理"列）
四、下周焦点（保留，增加"优先级"列）
五、新增成果（保留）
```

**2. "本周结论"生成逻辑**
- 后端规则驱动，非AI：
  - 如果 total_gmv_rate < 时间进度比例 → "GMV完成率低于时间进度，需加速"
  - 如果 riskCount > 0 → "当前有N个风险项目需关注"
  - 如果有 severe_warning → "XX指标严重预警"
- 前端展示为醒目摘要区

**3. 导出保留**
- PNG/Word/Markdown 导出逻辑完全保留
- 导出内容包含新增的结论区块

### 风险说明

| 改动 | 风险 | 说明 |
|------|------|------|
| 周报结论区块 | 低 | 后端在 generate 时追加 computed 字段，不改表结构 |
| 导出内容变更 | 低 | 追加内容，不删除原有内容 |

## 十、UI / 交互规范建议

### 状态色统一（系统级标准）

| 状态 | 颜色 | 适用模块 |
|------|------|---------|
| 完成/正常 | 🟢 #16A34A / Tag success | Project/MonthlyTask/Achievement |
| 进行中 | 🔵 #3B5AFB / Tag processing | Project/MonthlyTask |
| 风险/预警 | 🔴 #DC2626 / Tag error | Project/Performance |
| 严重 | 🔴 #DC2626 + 背景 #FEF2F2 | Performance |
| 未启动/待开始 | ⚪ #9CA3AF / Tag default | Project/MonthlyTask |
| 高优先级 | 🟠 #F59E0B / Tag warning | Achievement |

### 优先级排序规则（全局统一）

```
风险项 > 临期项(7天内) > 低进度项(<60%) > 长期未更新(>3天) > 普通项
```

### 时间标签统一

所有列表和卡片都要显示：
- `updated_at`：上次更新时间（相对时间：2小时前 / 昨天 / 3天前）
- 所属周期：Q2 / 4月 / 本周
- 截止时间：剩余N天 / 已逾期N天

### 交互规范

1. **就地更新**：进度条点击可编辑，文本字段点击可编辑
2. **筛选器**：每个列表默认按管理优先级排序，支持切换
3. **搜索**：全局搜索放在顶栏，可搜项目名/负责人/指标名
4. **抽屉替代跳转**：详情用Drawer展示，不跳新页面
5. **确认机制**：删除/批量操作需要二次确认

## 十二、需要新增或调整的数据字段、接口、前后端逻辑

### 新增后端接口

| 接口 | 方法 | 说明 | P阶段 |
|------|------|------|-------|
| `/api/dashboard/today-changes` | GET | 今日数据变更（读AuditLog） | P0 |
| `/api/dashboard/week-focus` | GET | 本周关注点（规则驱动） | P1 |
| `/api/dashboard/week-summary` | GET | 本周摘要统计 | P0 |
| `/api/projects/stale` | GET | 超N天未更新的项目 | P1 |
| `/api/projects/:id/quick-update` | PUT | 快速更新（progress/status/weekly_progress） | P1 |
| `/api/search?q=keyword` | GET | 全局搜索（跨表） | P2 |

### 现有接口调整

| 接口 | 调整 | 风险 |
|------|------|------|
| `GET /api/projects` | 增加 `sort=priority` 参数，支持管理优先级排序 | 低（向后兼容，不传sort则默认旧排序） |
| `GET /api/dashboard` | 增加 `today_changes` 和 `week_focus` 字段到返回 | 低（增量字段） |
| `POST /api/weekly-reports/generate` | 生成时追加 `week_conclusion` 和 `key_changes` 字段 | 低（增量字段） |

### 数据库字段（无新增表，只考虑新增虚拟计算字段）

不需要修改数据库表结构。所有新增能力都基于现有字段计算：
- `updated_at` → 推算"多久没更新"
- `due_date` → 推算"距截止几天"
- `progress_pct` + `status` → 推算"管理优先级"
- `AuditLog.changed_fields` → 推算"今日变化"
